# RTB 追踪 URL 体系详解

> 基于 honghu-ad-engine 参考实现 + Bolt 当前实现的对照分析

---

## 一、RTB 追踪的两条独立链路

RTB 竞价胜出后，追踪回调分为两套独立体系，各自服务不同主体：

| 链路 | 生成方 | 回调触发方 | 目的 |
|------|--------|-----------|------|
| 引擎侧追踪（/i、/c） | ADX/引擎自己 | 客户端 SDK | 引擎记录曝光/点击，作为计费和报表数据源 |
| DSP 侧追踪（aimps、aclks、nurl） | DSP 出价时返回 | 客户端 SDK / 媒体 | 通知 DSP 发生了曝光/点击，传递结算价 |

两类 URL 拼成数组一起下发给客户端，SDK 展示/点击广告时**依次回调数组中所有 URL**：

```
impressionUrls = [引擎/i, DSP展示追踪URL1, DSP展示追踪URL2, ...]
clickUrls      = [引擎/c, DSP点击追踪URL1, ...]
```

引擎排首位，保证自己的计数优先到达。

---

## 二、引擎侧追踪（我们已实现）

### 角色

引擎的 `/i` 和 `/c` 是**计费监测点**——记录"这条广告确实被用户看到/点击了"。

### 生成时机

竞价决策完成**之后**，BidService 组装响应时生成。此时所有值（bidid、adSourceId、settlePrice）已确定，无需宏替换。

### URL 结构

```
GET /i?bid=<bidid>&src=<adSourceId>&p=<settlePrice>&ts=<时间戳>&sig=<HMAC-SHA256>
GET /c?bid=<bidid>&src=<adSourceId>&p=<settlePrice>&ts=<时间戳>&sig=<HMAC-SHA256>
```

### 回调处理

| 端点 | 验签 | 记录 | 响应 |
|------|------|------|------|
| GET /i | HMAC 验签，失败返回 403 | 记录展示日志 | 200 + 1x1 透明 GIF |
| GET /c | HMAC 验签，失败返回 403 | 记录点击日志 | 302 重定向至落地页 |

### 为什么不需要宏替换

引擎追踪 URL 在竞价决策**之后**生成，此时结算价已确定。宏替换是为了解决"生成 URL 时结算价未知"的问题——这只发生在 DSP 侧（DSP 出价时还不知道自己会不会赢）。

## 三、DSP 侧追踪（TODO，待实现）

### 角色

通知 DSP："你赢了，结算价是多少"（nurl）以及"你的广告被展示/点击了"（aimps/aclks）。

### 宏替换的必要性

DSP 在出价阶段返回 nurl，此时**竞价尚未结束**，DSP 不知道：
- 自己会不会赢
- 最终结算价是多少

所以 DSP 在 URL 中放占位宏：

```
https://dsp.example.com/win?price=${AUCTION_PRICE}&bid=${AUCTION_BID_ID}
```

引擎竞价决策后，将宏替换为真实值再下发/回调。

### 标准宏（OpenRTB）

| 宏 | 含义 |
|---|---|
| `${AUCTION_PRICE}` | 结算价 |
| `${AUCTION_BID_ID}` | 竞价 ID |
| `${AUCTION_IMP_ID}` | 广告位 ID |
| `${AUCTION_CURRENCY}` | 货币代码 |
| `${AUCTION_SEAT_ID}` | 席位 ID |

### 参考项目的做法

honghu-ad-engine 更进一步：
- 各 DSP 有私有宏（OPPO: `$swp$`，同程: `__TC_AUCTION_PRICE__`）
- 引擎先将私有宏统一转换为内部宏（`__BID_PRICE_AES__`）
- 最后统一替换为 AES 加密后的价格

价格加密是为了防止链路上中间人截获明文结算价。

## 四、完整回调时序

```
┌─────────────┐       ┌─────────┐       ┌─────────┐       ┌─────────┐
│ 客户端 SDK  │       │  引擎   │       │  DSP    │       │  媒体   │
└─────┬───────┘       └────┬────┘       └────┬────┘       └────┬────┘
      │                    │                  │                  │
      │  展示广告          │                  │                  │
      ├───GET /i──────────▶│ 记录曝光日志     │                  │
      │◀──1x1 GIF─────────┤                  │                  │
      │                    │                  │                  │
      ├───GET dsp_imp_url─────────────────────▶ DSP 记录曝光     │
      │◀──200──────────────────────────────────┤                  │
      │                    │                  │                  │
      │  用户点击          │                  │                  │
      ├───GET /c──────────▶│ 记录点击日志     │                  │
      │◀──302 → 落地页────┤                  │                  │
      │                    │                  │                  │
      ├───GET dsp_clk_url─────────────────────▶ DSP 记录点击     │
      │◀──200──────────────────────────────────┤                  │
```

## 五、Bolt vs 参考项目对比

| 维度 | 参考项目（honghu-ad-engine） | Bolt 当前实现 |
|------|---------------------------|--------------|
| 参数保护 | AES 加密整个参数体 | HMAC 签名（参数明文+sig 防篡改） |
| 价格传递 | 加密价格（`__BID_PRICE_AES__`） | 明文价格（p 参数） |
| 宏替换 | 支持（各平台私有宏→统一宏→替换） | 未实现（TODO-03） |
| 日志投递 | Kafka（lj-imp-dsp、lj-clk-dsp） | log.info |
| 追踪种类 | 5 种（/i、/c、/dl、/cusi、/cusc） | 2 种（/i、/c） |
| DSP 侧 URL | 宏替换后拼入数组下发 | 未实现 |

## 六、术语表

| 术语 | 含义 |
|------|------|
| nurl | Win Notice URL，竞胜通知 |
| lurl | Loss Notice URL，竞败通知 |
| aimps | 展示追踪 URL 数组 |
| aclks | 点击追踪 URL 数组 |
| beacon | 1x1 像素追踪（img 标签请求） |
| 宏替换 | 将 URL 模板中的占位符（如 `${AUCTION_PRICE}`）替换为真实值 |
